# vtext 架构决策记录（ADR）

记录关键架构决策及其权衡分析，供后续开发参考。

---

## ADR-001：音频提取移至客户端

**决策：** ffmpeg 音频提取在客户端完成，server 只接收 WAV。

**原因：**
- 视频文件体积大（1小时视频可达数 GB），通过网络传输代价高
- WAV（16kHz mono）体积远小于原始视频
- ffmpeg 是常见系统工具，作为客户端依赖可接受

**权衡：**
- ✅ 大幅减少网络传输量
- ✅ server 职责更单一，无需 ffmpeg
- ❌ 客户端需要安装 ffmpeg

## ADR-002：大文件 zstd 压缩传输

**决策：** WAV ≥ 100MB 时，客户端用 zstd 压缩后上传，通过 form field `encoding: zstd` 告知 server。

**原因：**
- 不依赖 HTTP 层压缩能力，控制权在客户端
- zstd 压缩/解压速度快，压缩率优于 gzip
- 100MB 阈值平衡了压缩开销和传输收益

**权衡：**
- ✅ 显著减少大文件传输时间
- ✅ 协议简单，server 解压逻辑少量代码即可实现
- ❌ 客户端新增 `zstandard` 依赖

## ADR-003：异步任务 + SSE 进度推送

**决策：** `POST /transcribe` 立即返回 `job_id`，进度通过 `GET /jobs/{id}/stream` SSE 推送。

**原因：**
- 转写是长耗时操作（数分钟），同步 HTTP 连接不可靠
- SSE 比轮询实时性好，比 WebSocket 实现简单
- 命令行进度条需要连续的进度数据流

**权衡：**
- ✅ 客户端进度展示流畅，无轮询跳跃感
- ✅ 网络中断后可通过 `GET /jobs/{id}` 恢复状态再重连
- ❌ server 需维护任务状态，复杂度高于同步方案

## ADR-004：多进程 worker 并发模型

**决策：** 任务队列 + 多进程 worker，默认 worker 数 = CPU 核心数。

**原因：**
- whisper.cpp 是 CPU 密集型，多线程受 GIL 限制无法真正并行
- 多进程每个 worker 独占一个核心，充分利用多核服务器
- 队列满时返回 429 + 队列信息，由客户端自决是否等待

**权衡：**
- ✅ 真正的多核并行，吞吐量随核心数线性提升
- ✅ 队列满时快速失败，避免无限排队
- ❌ 多进程内存开销高于多线程（每个 worker 加载一份模型）

## ADR-005：模型切换策略

**决策：** 默认模型常驻内存，切换时等待当前队列清空后再切换。

**原因：**
- 模型加载代价高（large 模型十几秒），频繁切换严重影响吞吐
- 等队列清空再切换保证任务间不中断
- 每个请求可通过 `model` 参数指定，灵活性保留

**权衡：**
- ✅ 避免频繁切换导致的性能抖动
- ✅ 多 worker 场景下模型一致，结果可预期
- ❌ 切换时需等待队列清空，响应有延迟
